# Alpine.js 重构评估规范

## Why
当前项目使用 Next.js 15 + React 19 + framer-motion 技术栈，存在大量未使用的第三方依赖（52个UI组件、recharts、zod等），导致包体积较大。需要评估是否可以用 Alpine.js 重构，实现更轻量化、更快的加载速度。

## What Changes
- 评估当前项目的技术栈复杂度
- 分析 Alpine.js 重构的可行性
- 识别重构的风险和挑战
- 提供重构建议方案

## Impact
- Affected specs: 整体架构、前端技术栈
- Affected code: 所有业务组件 (13个)、UI组件库 (52个)

- **THEN** 识别所有已安装的依赖及其使用情况

## 评估结论

### 当前项目状态

| 维度 | 状态 | 说明 |
|------|------|------|
| **框架** | Next.js 15.5.2 + React 19.1.1 | 最新版本 |
| **动画库** | framer-motion | 12个组件使用，含高级特性 |
| **UI组件库** | shadcn/ui (52个) | 大部分未使用 |
| **状态管理** | useState | 简单，无复杂状态 |
| **SSR使用** | 低 | 大部分是客户端组件 |

### 依赖冗余分析

| 类别 | 已安装 | 实际使用 | 冗余率 |
|------|--------|----------|--------|
| @radix-ui 组件 | 32个 | 0个 | 100% |
| UI 组件文件 | 52个 | ~5个 | ~90% |
| recharts | 1个 | 0个 | 100% |
| zod | 1个 | 0个 | 100% |
| react-hook-form | 1个 | 0个 | 100% |

### Alpine.js 重构可行性

#### ✅ 适合重构的因素
1. **简单的状态管理** - 仅使用 useState，无复杂状态依赖
2. **展示型组件为主** - 大部分是静态展示 + 简单动画
3. **冗余依赖多** - 清理后可大幅减少包体积
4. **SEO 需求低** - 企业官网，内容相对静态

#### ⚠️ 重构挑战
1. **framer-motion 高级动画** - 使用了 useScroll, useTransform, useSpring, useInView 等高级特性
2. **主题切换** - 需要重新实现 next-themes 功能
3. **拖拽轮播** - Advantage.tsx 的拖拽功能需要替代方案

### 推荐方案

| 方案 | 描述 | 包体积减少 | 开发成本 | 动画效果 |
|------|------|-----------|----------|----------|
| **方案A: 完全重构** | Alpine.js + CSS动画 + 静态生成 | ~70% | 高 | 降级 |
| **方案B: 保留优化** | 移除未使用依赖，保留核心 | ~40% | 低 | 保持 |
| **方案C: 混合方案** | 静态HTML + Alpine.js + 保留复杂动画组件 | ~50% | 中 | 部分保持 |

### 最终建议

**推荐方案B：保留 React + 优化依赖**

理由：
1. framer-motion 动画效果是项目的核心体验，降级会影响用户体验
2. 清理未使用依赖即可获得显著的包体积优化
3. 开发成本低，风险小

如果追求极致性能：
- 选择方案A，但需要接受动画效果降级
- 或选择方案C，将复杂动画组件保留为 Web Component
